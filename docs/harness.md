# Harness Rules ガイド

本プロジェクトでは `.claude/rules/` に配置したルールファイルで Claude Code の振る舞いを制御している。
このドキュメントでは、claude-cookbooks 由来の5つのルールについて、それぞれの役割・解決する課題・得られるメリットを説明する。

## 全体像

```
.claude/rules/
├── agent-design-patterns.md     ← エージェント設計パターン
├── tool-use-strategy.md         ← ツール使用戦略
├── extended-thinking-strategy.md ← 拡張思考戦略
├── mcp-hook-integration.md      ← MCP・Hook 統合ルール
└── prompt-caching-cost.md       ← プロンプトキャッシュ・コスト最適化
```

## 1. agent-design-patterns.md — エージェント設計パターン

### 役割

サブエージェントを設計・追加する際の **パターン選択基準** を定義する。
5つの基本パターン（Chaining / Routing / Parallel / Orchestrator-Subagents / Evaluator-Optimizer）の使い分けを判断木で示し、既存のエージェント群との対応関係を明記している。

### 解決する課題

| 課題                 | 発生する状況                                                   |
| -------------------- | -------------------------------------------------------------- |
| エージェントの肥大化 | 1つのエージェントに複数責務を詰め込み、判断品質が低下する      |
| 自律ループの暴走     | 脱出条件なしのループが無限に回り、トークンとコストを浪費する   |
| パターンの不統一     | 開発者ごとに異なる設計思想でエージェントが増え、保守困難になる |

### メリット

- **設計の一貫性**: 新規エージェントもチェックリストに沿って設計するため、品質が安定する
- **既存資産の活用**: adversarial パイプラインや diversity-synthesis がどのパターンに該当するか明文化されているため、拡張・改修時に設計意図を見失わない
- **安全性**: 禁止事項（無限ループ、責務過多）が明示されているため、危険な設計を事前にブロックできる

## 2. tool-use-strategy.md — ツール使用戦略

### 役割

Claude Code がツールを呼び出す際の **判断基準・並列化ルール・コンテキスト予算管理** を定義する。
「どのツールを使うべきか」「並列 or 逐次か」「どれだけ読み込むか」を機械的に判断できるよう設計している。

### 解決する課題

| 課題               | 発生する状況                                                                 |
| ------------------ | ---------------------------------------------------------------------------- |
| 不適切なツール選択 | Bash で `cat` や `grep` を実行し、専用ツールの利点（レビュー体験向上）を失う |
| 逐次実行の非効率   | 独立した複数の読み込みを1つずつ順番に実行し、レスポンス時間が増大する        |
| コンテキスト圧迫   | 全ファイル読み込みや冗長な Grep 結果でコンテキストウィンドウを浪費する       |

### メリット

- **応答速度の向上**: 並列実行ルールにより、独立したツール呼び出しが同時に走る
- **コンテキスト節約**: JIT 取得・`offset`+`limit` 指定・`head_limit` 活用で、必要最小限の情報だけコンテキストに載せる
- **レビュー体験の向上**: 専用ツール使用により、ユーザーがツール呼び出しの意図を一目で理解できる

## 3. extended-thinking-strategy.md — 拡張思考戦略

### 役割

タスクの複雑度に応じた **Plan モードの使用基準と思考深度の調整方法** を定義する。
「いつ Plan モードを使うか」「思考とツール呼び出しをどう組み合わせるか」を3つのパターンで示している。

### 解決する課題

| 課題               | 発生する状況                                                           |
| ------------------ | ---------------------------------------------------------------------- |
| 計画なしの実装着手 | 複雑なタスクを考えずに開始し、途中で手戻りが発生する                   |
| 過剰な計画         | 単純な typo 修正にも Plan モードを起動し、不要なオーバーヘッドが生じる |
| 思考と行動の断絶   | 長く考えた後にまとめて行動し、中間結果によるフィードバックを逃す       |

### メリット

- **手戻りの削減**: 複雑度に応じた Plan モード使用で、方向性の誤りを早期検出できる
- **効率的な思考配分**: 単純タスクには軽い思考、重大タスクには深い推論と、リソースを適切に配分する
- **計画崩壊時のリカバリ**: 「3回再計画が必要ならタスク粒度が大きすぎる」という明確な基準で、泥沼化を防ぐ

## 4. mcp-hook-integration.md — MCP・Hook 統合ルール

### 役割

MCP サーバーの権限設計と Hook のライフサイクル管理の **設計基準・実装ガイド** を定義する。
既存の6つの Hook との整合表を含み、新規追加時のチェックリストを提供している。

### 解決する課題

| 課題            | 発生する状況                                                                 |
| --------------- | ---------------------------------------------------------------------------- |
| 権限の肥大化    | MCP ツールに過剰な権限を付与し、意図しない外部書き込みが発生する             |
| Hook の非冪等性 | 同じ Hook が複数回実行されると結果が変わり、予測不能な動作になる             |
| 責務の重複      | 新しい Hook が既存 Hook と同じことをチェックし、メンテナンスコストが増大する |

### メリット

- **セキュリティの担保**: 4段階の権限スコープ（Read-only → External write）で最小権限の原則を機械的に適用できる
- **既存基盤との整合**: 6つの既存 Hook の役割と変更時の注意が一覧化されているため、新規追加時の衝突を防げる
- **Chief of Staff パターン**: サブエージェント連携の標準構造が定義されているため、ワークフロー拡張時に設計判断が速くなる

## 5. prompt-caching-cost.md — プロンプトキャッシュ・コスト最適化

### 役割

Anthropic API 利用時の **プロンプトキャッシュ戦略とコスト最適化** のルールを定義する。
コストテーブル、モデル別最小トークン数、投機的キャッシュ、ScheduleWakeup との関係を網羅している。

### 解決する課題

| 課題                    | 発生する状況                                                                  |
| ----------------------- | ----------------------------------------------------------------------------- |
| API コストの浪費        | キャッシュ可能なコンテキストを毎回フル送信し、入力コストが10倍になる          |
| TTFT の遅延             | 大きなコンテキストを毎回再処理し、応答開始までの待ち時間が増大する            |
| ScheduleWakeup の非効率 | キャッシュ TTL（5分）を無視したスリープ設計で、不要なキャッシュミスが発生する |

### メリット

- **最大90%のコスト削減**: キャッシュ読み取りは0.1倍コスト。繰り返し呼び出しで劇的に効果が出る
- **応答速度の向上**: 投機的キャッシュで TTFT を最大91%削減。ユーザー体験が改善する
- **モデル選択の最適化**: 役割×モデル×キャッシュ戦略のテーブルで、コストとパフォーマンスのバランスを取れる

## ルールの適用タイミング

`.claude/rules/` に配置されたルールファイルは、Claude Code のセッション開始時に自動的にコンテキストに読み込まれる。開発者が明示的に参照する必要はなく、Claude Code は各ルールに従って判断・行動する。

### ルール間の関係

```
agent-design-patterns ──→ エージェント設計時に参照
        │
        ├── tool-use-strategy ──→ エージェント内のツール呼び出しに適用
        │
        ├── extended-thinking-strategy ──→ Plan モード判断に適用
        │
        └── mcp-hook-integration ──→ Hook / MCP 追加時に参照
                │
                └── prompt-caching-cost ──→ API 呼び出し最適化に適用
```

### 更新方針

- 各ルールは **claude-cookbooks の更新** に追従して改訂する
- 新しいエージェントやワークフローを追加した際は、該当ルールの「既存エージェント例」を更新する
- ルール同士の矛盾が見つかった場合は、上位ルール（agent-design-patterns）を優先する
